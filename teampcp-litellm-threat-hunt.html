How TeamPCP Backdoored the World's Most Popular AI Gateway — and Why GovCon Environments Are Exposed

A cascading supply chain compromise starting with a single Trivy CI/CD token culminated in the backdooring of LiteLLM on PyPI — a package with 3.4 million daily downloads used across AI-integrated government contractor environments.

01

Executive Overview

🚨
Immediate Action Required

If your organization installed or upgraded litellm via pip on March 24, 2026 between 10:39 UTC and 16:00 UTC, assume full compromise. Rotate all secrets, revoke credentials, and initiate IR procedures immediately.

On March 24, 2026, threat actor TeamPCP published two trojanized versions of the litellm Python package to PyPI — versions 1.82.7 and 1.82.8. LiteLLM is an open-source AI gateway used by developers and organizations to route requests across 100+ LLM APIs (OpenAI, Azure, Anthropic, Bedrock, etc.).

The attack did not exploit a vulnerability in LiteLLM's code. Instead, TeamPCP compromised the CI/CD pipeline that publishes LiteLLM to PyPI — specifically by first poisoning Aqua Security's Trivy container scanner, which ran inside LiteLLM's GitHub Actions build process. This allowed the attacker to silently steal the PyPI publish token and inject a multi-stage credential-stealing backdoor into the published package.

For government contractors and GCC environments, the risk is acute: LiteLLM is a common integration layer for AI-powered applications, internal copilots, and LLM proxy deployments that may have access to cloud credentials, API keys, Kubernetes configs, and sensitive environment secrets.

🎯

Who Is at Risk?

Organizations that pip-installed or upgraded litellm on March 24, 2026. CI/CD pipelines, Docker-based deployments without pinned deps, developer workstations, and self-hosted LLM proxy servers.

💀

What Can the Malware Do?

Harvest cloud credentials, SSH keys, K8s secrets, .env files, crypto wallets, and CI/CD tokens. Deploy privileged Kubernetes pods. Install a persistent systemd backdoor for ongoing access.

🏛️

GovCon Specific Risk

Exposure of GCC tenant API keys, Azure Government credentials, CAC-adjacent service account tokens, and FedRAMP boundary secrets. Potential for lateral movement into controlled environments.

🛡️

Who Was NOT Impacted

Organizations running the official LiteLLM Proxy Docker image were not affected — it pins dependencies in requirements.txt and does not pull from PyPI at runtime.

02

Attack Timeline — The TeamPCP Campaign

This was not an isolated incident. TeamPCP executed a cascading five-day campaign targeting open-source security tooling used in developer CI/CD pipelines. LiteLLM was the third major victim.

March 19, 2026 — Day 1

Stage 1: Aqua Security Trivy Compromised

TeamPCP obtained a GitHub PAT (personal access token) belonging to an Aqua Security maintainer. Using it, they forcibly overrode 76 of 77 version tags in aquasecurity/trivy-action and all 7 tags in aquasecurity/setup-trivy, redirecting trusted CI tags to malicious commits containing a credential harvester. Any workflow pinning aquasecurity/trivy-action@v0.x now executed attacker-controlled code.

March 23, 2026 — Day 5

Stage 2: Checkmarx GitHub Actions & VS Code Extensions Compromised

Credentials harvested from Trivy CI runs were used to compromise Checkmarx's ast-github-action and multiple OpenVSX VS Code extensions. An identical credential stealer was injected, exfiltrating secrets to typosquat domain checkmarx.zone. Over 66 npm packages were subsequently compromised via stolen npm tokens, introducing CanisterWorm — the first publicly documented npm malware to use an Internet Computer Protocol (ICP) blockchain canister as a C2 dead-drop.

March 24, 2026 — Day 6 — 10:39 UTC

Stage 3: LiteLLM's CI/CD Pipeline Poisoned — PyPI Token Stolen

LiteLLM's GitHub Actions pipeline ran Trivy as part of its build process, pulling from apt without a pinned version. The compromised Trivy action executed, dumping Runner.Worker process memory and harvesting the PYPI_PUBLISH token from the GitHub Actions environment. TeamPCP now had direct publish access to the litellm PyPI package.

March 24, 2026 — 10:39 UTC & 10:52 UTC

Malicious litellm 1.82.7 and 1.82.8 Published to PyPI

Within 13 minutes, two backdoored versions were live on PyPI. Version 1.82.7 embedded a base64-encoded payload in litellm/proxy/proxy_server.py. Version 1.82.8 escalated persistence by adding a litellm_init.pth file to site-packages, triggering the payload on every Python interpreter startup — no import required.

March 24, 2026 — ~16:00 UTC

PyPI Quarantines Packages — ~3 Hour Exposure Window

PyPI quarantined both versions approximately 3 hours after publication. The last known-clean version is litellm 1.82.6 (published March 22, 2026). CVE-2026-33634 was assigned with a CVSS4B score of 9.4.

March 25–26, 2026 — Ongoing

TeamPCP Continues — Telnyx PyPI SDK Compromised

The campaign did not stop with LiteLLM. Malicious versions of the Telnyx Python SDK (4.87.1, 4.87.2) were published. TeamPCP introduces a Windows-specific payload variant that downloads hangup.wav from C2, decodes an XOR-obfuscated executable from audio frames, and drops it as msbuild.exe in the Windows Startup folder.

Technical Analysis — Three-Stage Malware

🔬
Analyst Note

Both compromised versions share the same core payload. Version 1.82.8 adds a persistence mechanism via Python's .pth file injection, making it significantly more dangerous as it fires on every Python invocation — even in environments that never directly import litellm.

Stage 1 — Credential Harvester

On execution, the payload sweeps the host filesystem and environment for credentials across more than 50 known paths:

Target TypePaths / Patterns
SSH Keys~/.ssh/id_*, ~/.ssh/config
Environment Variables.env, .envrc, process env dump
AWS Credentials~/.aws/credentials, ~/.aws/config
GCP / Azure~/.config/gcloud/, ~/.azure/
Kubernetes~/.kube/config, SA tokens, cluster secrets
Docker~/.docker/config.json
Database.pgpass, ~/.my.cnf, connection strings
Crypto WalletsWallet files, seed phrase patterns
CI/CD TokensGitHub PAT, npm, GitLab, CircleCI
Shell History~/.bash_history, ~/.zsh_history

Stage 2 — Kubernetes Lateral Movement

If a Kubernetes service account token is present, the malware:

  • Reads all secrets across all namespaces
  • Enumerates all cluster nodes
  • Deploys a privileged alpine:latest pod to every node in kube-system
  • Mounts the host filesystem (/) into the pod
  • Installs a persistent backdoor via the mounted host
⚠️
K8s Backdoor Pods

Look for pods named node-setup-* in the kube-system namespace. These are attacker-controlled persistent backdoors.

Stage 3 — Systemd Persistence

A user-level systemd service (sysmon.service) is installed at ~/.config/systemd/user/. It polls checkmarx.zone/raw on a schedule to download and execute additional binaries — enabling ongoing C2 access after initial credentials have been rotated.

Persistence Mechanism — The .pth File Trick

Python's site-packages mechanism automatically executes any .pth file found at interpreter startup. By inserting litellm_init.pth into site-packages, version 1.82.8 ensures payload execution on every Python invocation system-wide — including in virtual environments. This is a known but underappreciated Python persistence technique that bypasses most application-layer monitoring.

Python — .pth Persistence Mechanism How litellm_init.pth triggers payload execution 
# site-packages/litellm_init.pth  (34,628 bytes)
# Python evaluates import lines in .pth files at every interpreter startup
# No explicit "import litellm" needed — fires for ALL python invocations

# Decodes and executes the embedded base64 payload

Threat Hunting Queries & Detection

Use the following queries to hunt across your environment. Prioritize systems where Python is used in production, CI/CD pipelines, and any deployment running an LLM integration layer.

Host-Based — File System Checks

Bash — Compromise Detection Script Run on every potentially affected host 
#!/bin/bash
# TeamPCP / LiteLLM Compromise Detection
# Run as: bash litellm-hunt.sh | tee hunt-results.txt

echo "[*] Checking litellm version..."
pip show litellm 2>/dev/null | grep "Version:"

echo "[*] Scanning for malicious .pth file..."
find / -name "litellm_init.pth" 2>/dev/null && echo "ALERT: .pth FOUND"

echo "[*] Checking for sysmon backdoor artifacts..."
test -f ~/.config/sysmon/sysmon.py && echo "ALERT: sysmon.py FOUND"
test -f ~/.config/systemd/user/sysmon.service && echo "ALERT: sysmon.service FOUND"

echo "[*] Checking systemd user services..."
systemctl --user list-units --type=service 2>/dev/null | grep sysmon

echo "[*] Checking for attacker K8s pods (if kubectl available)..."
kubectl get pods -n kube-system 2>/dev/null | grep "node-setup-"

echo "[*] Checking network connections to C2..."
ss -tnp | grep "83.142.209"
netstat -tnp 2>/dev/null | grep "83.142.209"

echo "[*] Checking DNS resolution of C2 domains..."
host models.litellm.cloud 2>/dev/null | grep "83.142.209" && echo "ALERT: C2 domain resolved"
host checkmarx.zone 2>/dev/null | grep "83.142.209" && echo "ALERT: C2 domain resolved"

echo "[*] Checking pip audit..."
pip-audit 2>/dev/null | grep litellm

echo "[*] Hunt complete."

Incident Response — GovCon & GCC Playbook

🚨
If Compromise is Confirmed

Treat the entire host as fully compromised. Do not attempt to clean and continue. Isolate, capture forensic artifacts, then rebuild from a known-good image. All secrets accessible on the host must be considered exfiltrated.

  • Identify Exposure Window

    Determine if any host pip-installed or upgraded litellm on March 24, 2026 between 10:39 UTC and 16:00 UTC. Check pip history logs, CI/CD build logs, Docker build caches, and requirements.txt lock files.

  • Isolate Affected Systems

    Remove affected hosts from network segments. For GCC environments: quarantine from GCC tenant, revoke associated managed identities, and disable service principals before investigation.

  • Rotate ALL Secrets on Affected Hosts

    This is non-negotiable. Rotate: LLM API keys (OpenAI, Azure, Anthropic, Bedrock), AWS/GCP/Azure credentials, SSH key pairs, Kubernetes service account tokens, database passwords, .env values, GitHub PATs, npm tokens, HashiCorp Vault tokens, and any CI/CD secrets accessible in the environment.

  • Remove Backdoor Artifacts

    Remove litellm_init.pth from all Python site-packages, stop and disable sysmon.service (systemctl --user stop sysmon.service && systemctl --user disable sysmon.service), delete ~/.config/sysmon/, and delete attacker K8s pods (kubectl delete pod -n kube-system -l app=node-setup).

  • Upgrade to Clean litellm Version

    Uninstall the compromised version and reinstall: pip uninstall litellm -y && pip install litellm==1.82.6 or the latest verified clean version. Pin versions in requirements.txt and use hash verification in production.

  • Review Kubernetes Cluster Integrity

    Enumerate all pods in kube-system for unauthorized containers. Review RBAC bindings for anomalous service account permissions. Consider cluster credential rotation if a K8s config was present on any compromised host.

  • GovCon / GCC Notification Obligations

    Assess FISMA/CMMC reporting obligations if CUI-adjacent systems were exposed. GCC tenants should notify their ISSO and review FedRAMP boundary impact. Review DFARS 252.204-7012 cyber incident reporting requirements (72-hour window to DoD).

  • Harden CI/CD Pipeline Going Forward

    Pin all third-party GitHub Actions to commit SHAs (not tags). Use OIDC instead of long-lived PATs for PyPI publishing. Enable PyPI trusted publishers. Implement supply chain security policies via SLSA or in-toto attestations.
